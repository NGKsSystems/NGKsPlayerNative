# Ancillary Screens Phase 2: Old vs New Inventory

This inventory compares the donor NGKsPlayer ancillary implementation in `_artifacts/NGKsPlayer_donor` against the current NGKsPlayerNative ancillary implementation.

What the old app still had that the new app lacked before the Phase 2.6 pass:
- Crowd Requests had a real local listener, guest join page, live request ingestion, voting, and persisted operator state through the Electron host.
- Streaming had real service-controller plumbing for authentication/search/recommendations/playlists rather than a local-library proxy.
- Hardware had Web MIDI-driven controller discovery and a dedicated MIDI integration surface.
- Crowd Requests also exposed feedback, tips, and connection counts that are still not present in the native port after the Phase 2.6 handoff pass.

What was only visual before this pass:
- Crowd Requests start/stop state, join URL, queue count, queue contents, and operator actions in the native screen were all widget-local state with no backend.
- Streaming service cards in the native screen are still visual connect/disconnect toggles.
- Hardware DVS/MIDI/settings sections remain mostly informational in the native screen.

What is wired and actually works after this pass:
- Crowd Requests now runs through a real Python local HTTP server managed by the native app, with SQLite persistence, library-backed search from `data/runtime/library.json`, duplicate suppression, per-client request/vote cooldowns, request submission, voting, operator accept/reject/remove/clear actions, join URL reporting, QR generation, a richer mobile guest page, now-playing sync, near-live queue polling, native QR display/export controls, and real deck A or deck B handoff verification for matched local files.
- Streaming search and deck loading remain real in the native app, but they are backed by the imported local library rather than external service APIs.
- Live Streaming Setup remains a real native broadcast preview window with software/theme/resolution selection and now-playing refresh.
- Hardware Integration remains partly real through engine-visible audio-device scan and active-device switching.

Overall parity readout:
- Hardware Integration: partial parity with one real native improvement (actual engine device switching) but still missing real MIDI/DVS depth.
- Crowd Requests: request acceptance is now part of the real DJ workflow because the native app only records `HANDED_OFF` after the target deck reports the requested file and decode readiness. It still lacks donor extras such as feedback/tips, live connection counts, and a richer diagnostics surface.
- Streaming Music Services: visual/operator parity is decent, but platform integrations are still mostly absent.
- Live Streaming Setup: operator flow is largely preserved, but it remains a local preview workflow instead of deeper broadcaster integration.

What Crowd Requests now exceeds relative to the donor:
- The native queue now exposes explicit `HANDOFF_FAILED` and `HANDED_OFF` states rather than collapsing everything into a generic accepted bucket.
- The native queue now persists audited handoff detail text and local anti-spam policy outcomes alongside request history.
